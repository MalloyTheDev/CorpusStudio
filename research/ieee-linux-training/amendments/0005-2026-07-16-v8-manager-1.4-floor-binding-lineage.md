- **Amendment id:** `cs-ieee-linux-training-amendment-0005`
- **Study:** `cs-ieee-linux-training-v1`
- **Base protocol version:** 1.0.0
- **Effective protocol version:** 1.5.0 (supersedes 1.4.0)
- **Status:** prospective (authored before any v8 wheel build, environment, plan, or run)
- **Authored:** 2026-07-16
- **Analysis role:** primary

This amendment is **append-only**. It does not edit `PROTOCOL.md`, `EXPERIMENT_MATRIX.yaml`, amendments
0001-0004, effective matrices 1.1.0-1.4.0, or `RESERVED_IDENTITIES.v1`..`.v4` in place. It adds effective

narrative + manifest. The prior amendment 0004 is bound by exact raw-file hash in the manifest
`supersedes` block, so the amendment chain stays ordered and 0004 stays provably byte-frozen.

## Why this amendment exists

The v7 0.5B bring-up (effective matrix 1.4.0, worker wheel
`090f879b46d52e8c33c96fad8aeb61a41c320d44c20bd189dabfd5be606479b2`, worker source `21aa81d97ff752709fd4d03791288c1bb76a2339`)
produced **`V7_MATH_AND_FLASH_THROUGHPUT_PASS`**: both matched arms completed 12 QLoRA optimizer steps
with decreasing loss, admitted a clean adapter, measured `NATIVE_SAFE`, and - unlike v6 - recorded
**valid** token throughput (math `run-019f6956-d55c-7684-8c7a-9ebb7bad7a04` ~104 tok/s, flash
`run-019f6966-1c87-71e4-8c30-6fc5fa085caf` ~108 tok/s), with `scientific_throughput_complete=true` on
both arms. Those v7 identities are now fully instantiated and are reserved here.

Two subsequent, merged control-plane corrections change the v8 **wheel artifact identity** and the
**Environment Manager lock generation**, which forces a fresh v8 wheel and sealed environment pair:

1. **Exact per-lineage source-floor binding (PR #471, merged `4bab1de6a844acfb7443344880c5a82c1ac8e872`).**
   `required_git_ancestor` is now an exact, per-lineage plan input supplied to
   `env-plan --required-git-ancestor`, sealed into the `DependencyResolution`/confirmation hash, admitted
   by byte-equality against the worker wheel's EMBEDDED floor at `env-create`, and recorded in the
   `EnvironmentLock`. It is not a recipe constant and not a permanent historical minimum. The v8 wheel
   must EMBED this exact floor and be built with `build-worker-wheel --required-ancestor 4bab1de6a844acfb7443344880c5a82c1ac8e872`.
2. **Build-provenance `source_commit` emission (the v7 follow-up).** The worker build-provenance
   generator must emit the canonical `source_commit` key (v7 embedded it only under a prohibited alias).
   A v8 wheel's embedded `BUILD_PROVENANCE.json` therefore differs from v7's, changing the wheel
   content hash and forcing a fresh wheel identity.

## Lineage-change classification (wheel/artifact identity, not worker execution)

The two classes of reason for a fresh lineage are kept **separate and are not conflated**:

- `NEW_WHEEL_AND_ENVIRONMENT_LINEAGE_REQUIRED` (**true** here): the v8 wheel content hash changes (the
  emitted `source_commit` and embedded floor in `BUILD_PROVENANCE.json`), and the Environment Manager
  lock generation changes (1.3.0 -> 1.4.0, a new additive `EnvironmentLock` field). Sealed environments
  are pinned to a wheel content hash and a manager lock generation, so both facts force a fresh v8 wheel
  and environment pair.
- `WORKER_EXECUTION_CHANGE_REQUIRED` (**false** here): none of the merged corrections change the worker
  execution closure - the `worker.py` -> `supervisor.py` -> `artifacts.py` -> `runners.py` ->
  `training/trainer.py` runtime bytes that actually execute the optimizer steps. PR #471 touched
  control-plane admission (`environment_manager.py`, `environments.py`, `cli.py`) and the shared contract
  (an additive optional `required_git_ancestor` that defaults to null with no runtime behavior change),
  and the build-provenance generator is a build-time artifact writer, not run-time training code.

This is the honest distinction from the **v6 -> v7** transition, which *was* a genuine worker-execution
change (the `training_step` token observer inside `training/trainer.py`). For v8 the correct reason code
is therefore `worker-artifact-identity-and-manager-lock-generation-changed-fresh-v8-required`, not a
worker-execution-change claim.

## Environment Manager 1.4.0 (the version decision)

The Environment Manager mints one minor version per additive lock/evidence generation: 1.0.0 (introduce)
-> 1.1.0 (flash timing/thermal evidence) -> 1.2.0 (installed-file tree digest) -> 1.3.0 (successful-run
evidence). PR #471 added `required_git_ancestor` to the `EnvironmentLock` (a new additive lock field) and
changed worker-recipe admission behavior. Under the established convention this warrants **1.4.0**, and
this amendment declares `environment_admission.manager_version_exact = "1.4.0"` so the sealed matrix and
the implementation move together (the tested `manager_version_exact == MANAGER_VERSION` invariant). The
change is provably additive: `recipe_digest`, `resolution_digest`, and `_lock_digest` pop the field when
null, so every non-worker recipe and every pre-existing lock/resolution retains a byte-identical seal.
Historical matrices 1.1.0-1.4.0, amendments 0001-0004, `RESERVED_IDENTITIES.v1`..`.v4`, and all legacy
manager-1.3 locks/environments/evidence remain unchanged historical artifacts; manager 1.4.0 does not
rewrite or reclassify any of them.

## Two distinct arms (do not conflate them)

This amendment keeps two arms strictly separate:

- **Arm A - the bring-up validation tuple (unchanged).** The bounded Qwen2.5-0.5B / seq-256 QLoRA smoke
  is the tuple used to *validate that a fresh worker lineage is sound*. It is unchanged from 1.4.0 and is
  what a v8 pass re-demonstrates on the fresh wheel/environment pair.
- **Arm B - the 7B feasibility ladder (newly preregistered, non-paper).** A separate, non-paper
  characterization arm over Qwen2.5-7B-Instruct across sequence-length rungs; see below. Arm B is never a
  primary paper cell and never substitutes for Arm A.

### Arm A: what 1.5.0 does NOT change (the scientific tuple is preserved)

Qwen2.5-0.5B bring-up model; chat fixture; sequence length 256; microbatch 1; gradient accumulation 1;
12 optimizer steps; QLoRA r16 / alpha 32 / dropout 0.05 / bias none / all-linear; NF4 + double
quantization; BF16 compute; FP32 adapter weights and materialized gradients; `adamw_torch`;
checkpoint-free; no offload; no compile; no truncation; no packing; forced math versus forced flash. The
token-observation boundary, null-not-zero rule, rate-equals-observed-tokens-over-duration rule, and the
`scientific_throughput_complete` / `paper_performance_complete` promotion gates from 1.4.0 are retained.

## Arm B: seven-B native-Linux feasibility ladder (separate non-paper arm)

This amendment defines a separate, non-paper feasibility arm `seven_b_native_linux_feasibility_ladder`.
It is `classification: non-paper-feasibility`, `is_primary_paper_cell: false`, and
`satisfies_three_trial_characterization_matrix: false` - it never substitutes for a primary paper cell
and never satisfies the three-trial characterization matrix.

**Model reference (resolved, not repository-shaped).** The arm binds `model_id: "qwen2.5-7b-instruct"`,
which resolves to exactly one `models[].id` in the base matrix, with
`model_source_repository: "Qwen/Qwen2.5-7B-Instruct"` matching that model's `source_repository`. A
repository-shaped `model_id` (one containing `/`) or an id that does not resolve to exactly one model is
refused by the validator.

**Feasibility fixture (its own identity contract, not the private corpus).** The arm runs against a
separate, immutable, small, license-clear chat fixture `cs-ieee-linux-7b-feasibility-chat-fixture-v1`, declared under
`feasibility_fixture` with `is_primary_private_corpus: false`. Its `content_sha256`,
`rendered_examples_sha256`, `tokenizer_content_sha256`, and `chat_template_sha256` are each
`required-before-planning`; `license_evidence_required: true`; `fixed_row_order: true`; `packing: false`;
`truncation: false`. **These bindings belong to the feasibility fixture, not the primary private corpus**,
which is untouched. No fixture content is read or frozen in this amendment.

**Fixed ladder configuration (exact).** `fixed_ladder_configuration` binds rungs [512, 1024, 2048, 3072,
4096] in ascending order, microbatch 1, gradient accumulation 1, 12 bounded optimizer steps, offload
none, zero automatic workload retries, math runs first-once, flash runs at-most-once. The validator binds
each value exactly (not by prose).

**Flash eligibility (grounded in the real FailureRecord taxonomy + stage).** `flash_condition:
run-after-math-success-or-mapped-clean-math-terminal-taxonomy-and-stage`. Flash runs when EITHER math
succeeds OR, after a math failure, the concrete terminal `FailureRecord`
(`corpus_studio.platform.contracts.FailureRecord`; evidence fields `taxonomy` and `stage`) matches the
top-level `math_terminal_flash_eligibility` mapping, under confirmed forced-math with no fallback and
after every clean-failure precondition held (shared model/tokenizer/fixture preparation passed, declared
math kernel selected with no fallback, evidence remained valid, clean process termination, GPU memory
released, environment health and drift checks passed). The mapping is grounded in the existing enums
(`FailureTaxonomy`, `StageMarker`) and is fail-closed:

- **Eligible:** `OOM` at `forward` or `backward`; `KERNEL_STALL` at `forward` or `backward`. So a math
  OOM at the forward/backward pass that flash would survive is not a false negative - flash still runs.
- **`OOM` at `model_load`, `optimizer_created`, `export`, or any other shared / non-forward-backward
  stage:** `default_action: withhold-flash-and-stop`.
- **Generic `TIMEOUT`:** `timeout_decision: withhold`. `FailureRecord` localizes a failure only by
  taxonomy and stage; its finest failure stage is the forward/backward pass, and no machine-readable
  field proves a `TIMEOUT` occurred within the active math attention execution. A generic `TIMEOUT` is
  therefore not conclusively kernel-specific and is withheld; broadening this would require a contract
  change adding a machine-readable math-attention-execution marker.
- **Every unknown, missing, or unmapped taxonomy/stage combination:** `unmapped_combination_action:
  NOT_RUN-stop-fail-closed`.

This is a **scheduling** eligibility decision (is flash worth trying at this rung?),
`decision_is_scheduling_eligibility_not_math_kernel_causation: true` - never a claim that the math kernel
caused the failure. The mapping declares the full `known_failure_taxonomy` and `known_stage_markers`
snapshots of the live enums (the engine test binds them to the contracts exactly), and the invented
`kernel_specific_oom` / `kernel_specific_timeout` classes are removed. A withheld flash is
`withheld_flash_status: NOT_RUN`.

**Rung result definitions (separate).** `kernel_success` = one kernel satisfies every
`rung_success_requires` condition. `rung_success` = at least one executed kernel succeeds.
`matched_pair_success` = both math and flash succeed. `sequence_4096_feasibility_claim` = at least one
kernel succeeds at rung 4096. A flash failure never erases a valid math success; a math failure followed
by a valid flash success is a rung success but not a matched-pair success.

**Progression and stopping.** Advance to the next longer rung only after rung success; stop after a rung
where no kernel succeeds; all longer rungs become `NOT_RUN_PRIOR_RUNG_NO_SUCCESS`; no result is imputed to
any longer rung or otherwise; a shared-path failure stops the ladder immediately; every terminal result
is preserved exactly.

**Per-kernel success (exact, nothing weaker).** A kernel counts as a success only with all of: exactly 12
optimizer steps; finite loss at every step; the forced declared kernel with no fallback; positive token
evidence; changed adapter state; an admitted artifact; complete telemetry; a measured fit; and clean GPU
release (`rung_success_requires`). Sequence length 4096 is held to this identical set. This amendment
authorizes only the preregistration; it downloads no model, reads no fixture, and executes no rung.

## Preregistration boundary (unchanged)

Feasibility characterization remains distinct from the gated full-training phase over the ~500-output
corpus; that phase is not authorized here. No 7B model is loaded, no wheel is built, no environment is
created, and no rung is executed by this amendment.
